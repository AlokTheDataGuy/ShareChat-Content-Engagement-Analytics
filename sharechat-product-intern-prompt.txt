## Project Overview

**Name:** ShareChat Creator & Content Engagement Analytics

**Business problem:** How do we measure and improve user engagement, retention, and creator ecosystem health on a multi-language short-video platform — and what product levers should the team pull?

Three analytical questions fall out of this:
1. **User retention & engagement** — who stays, who churns, and why? (DAU/WAU/MAU, D1/D7/D30 retention, session depth)
2. **Content & creator performance** — which content types, languages, and creator tiers drive the platform?
3. **Monetization signals** — which user segments show highest ad/commerce intent?

## Synthetic Dataset Specifications

Use `Faker` + `NumPy` with a fixed random seed (`42`) for reproducibility. Build the data as a **star schema** — this is itself a talking point ("I modeled this the way a real Redshift warehouse would be structured at ShareChat").

**Scale:** 50,000 users, 5,000 creators, 100,000 posts, **~2 million events** over 90 days.

### Dimension: `dim_users` (50,000 rows)
- `user_id` (e.g., U_0000001)
- `signup_date` (spread across ~2 years, with more recent users than old)
- `signup_language` — weighted: Hindi 35%, Telugu 12%, Tamil 10%, Bhojpuri 8%, Marathi 7%, Bengali 7%, Kannada 6%, Malayalam 5%, Gujarati 4%, Punjabi 3%, Odia 2%, Assamese 1%
- `city_tier` — Tier-1/Tier-2/Tier-3/Tier-4 (roughly 15/25/35/25 split)
- `age_bucket` — 13-17, 18-24, 25-34, 35-44, 45+ (skewed young)
- `gender` — M/F/Other
- `device_type` — Android-Low / Android-Mid / Android-High / iOS (~60/25/10/5 — ShareChat is heavily Android)
- `acquisition_channel` — Organic/Paid/Referral/Preinstall
- `experiment_group` — control/variant (50/50 split for the simulated A/B test)

### Dimension: `dim_creators` (5,000 rows)
- `creator_id` (e.g., C_000001)
- `creator_signup_date`
- `primary_language` (same distribution as users)
- `follower_count` — **power law distributed** (most have <1K, few have millions)
- `creator_tier` — derived from follower_count: Nano (<1K), Micro (1K-10K), Mid (10K-100K), Macro (100K-1M), Mega (>1M)
- `content_category` — Comedy/Devotional/News/Music/Dance/Education/Lifestyle/Gaming/Regional-Drama
- `is_verified` — boolean (only ~3%)

### Dimension: `dim_content` (100,000 rows)
- `post_id`
- `creator_id` (foreign key — skewed so Mega creators post more)
- `post_date` (last 90 days)
- `content_type` — ShortVideo/Image/Text/LiveStream (weighted heavily ShortVideo)
- `language`
- `duration_seconds` (for videos; 15-90 sec typical)
- `has_music` — boolean
- `hashtag_count` — 0-15

### Dimension: `dim_date` (generate for 2 years)
- `date`, `day_of_week`, `week_num`, `month`, `is_weekend`
- `is_festival` — boolean, with `festival_name` — tag real Indian festival dates: Diwali, Holi, Eid-ul-Fitr, Eid-ul-Adha, Dussehra, Raksha Bandhan, Ganesh Chaturthi, Onam, Pongal, Navratri, Christmas, New Year

### Fact: `fact_sessions` (~500K rows, last 90 days)
- `session_id`
- `user_id`
- `session_start`, `session_end`
- `session_duration_sec`
- `posts_viewed`, `posts_liked`, `posts_shared`, `posts_commented`
- `device_type`
- `app_version` (e.g., 14.5.0)

### Fact: `fact_engagement_events` (~2M rows)
- `event_id`
- `user_id`
- `post_id`
- `creator_id` (denormalized for query performance — mention this in the schema doc as a real warehousing pattern)
- `event_type` — view/like/share/comment/follow/skip/report (views dominant)
- `event_timestamp`
- `watch_duration_sec` (only meaningful for video views)
- `scroll_velocity` — numeric, proxy for engagement quality

### Fact: `fact_ad_impressions` (~300K rows)
- `impression_id`
- `user_id`
- `ad_id` (synthetic)
- `impression_timestamp`
- `ad_category` (Ecommerce/Gaming/Fintech/FMCG/Travel/Education)
- `was_clicked` (boolean, realistic ~3% CTR)
- `was_converted` (boolean, ~10% of clicks)
- `revenue_inr` (only when clicked/converted)

### Realistic Behavioral Signals (must be present, these power the insights)
- **Retention curve** — classic social app shape: ~40% D1, ~20% D7, ~8% D30 retention
- **Language effects** — Hindi/Bhojpuri highest volume; Marathi/Bengali higher engagement depth (more likes/shares per view)
- **City tier pattern** — Tier-3/4 users have longer sessions, Tier-1 users convert better on ads
- **Festival spikes** — DAU 2-3x normal on major festival days; engagement also spikes
- **Weekend effect** — ~30% longer sessions on Sat/Sun
- **Creator power law** — top 1% of creators generate ~40% of engagement (Pareto)
- **Device effect** — low-end Android has higher crash/skip rates
- **A/B test lift** — variant group should have ~5-8% higher session duration and ~3-5% higher D7 retention than control, statistically detectable with proper testing

### Intentional Data Quality Issues (lighter than Citi project — for this role, it's about noticing them, not cleaning them)
- ~0.5% duplicate event rows
- A few sessions with `session_end < session_start` (flag in data_quality_checks)
- ~1% of events with null `watch_duration_sec` where event_type='view' (realistic — instrumentation gaps)
- Handful of test/internal user_ids (e.g., containing "TEST_") to be filtered

Save all tables as CSVs in `data/raw/`.

## Script-by-Script Specifications

### `src/01_generate_data.py`
Generates all dimension and fact tables with the dependencies and behavioral signals described. Heavily commented — each signal should have a comment explaining what real-world pattern it simulates. Output CSVs to `data/raw/`.

### `src/02_simulate_api_fetch.py`
Simulates fetching engagement events from an "internal API endpoint." Build this to **actually look like real API-fetching code**: uses `requests` style patterns (even if it's just reading a local CSV), handles pagination in a loop, retries on simulated failures, rate-limits, logs progress. This demonstrates the "scripting to fetch from API endpoints" skill in the JD. Output is a refreshed version of `fact_engagement_events.csv`. Include a docstring explaining the pattern would extend to real API endpoints with auth headers, etc.

### `src/03_build_warehouse.py`
Loads all CSVs into SQLite at `data/warehouse/sharechat_warehouse.db`. Creates the star schema properly with primary keys, foreign keys, and indexes on join columns (`user_id`, `creator_id`, `post_id`, `date`). Include a section at the top of the script as a comment explaining how this would differ on real Redshift (DISTKEY, SORTKEY choices).

### `src/04_data_quality_checks.py`
Runs sanity checks and prints a report:
- Row counts per table (sanity check they match expected scale)
- Null rate per column
- Referential integrity (every `user_id` in facts exists in `dim_users`)
- Date range validity (no future dates, no sessions with negative duration)
- Duplicate detection
- Flags the intentional quality issues for the analyst to notice

Output to console and also save as `data/warehouse/data_quality_report.txt`.

## 15 Advanced SQL Queries

Every query file must start with a comment block:
```sql
-- ============================================================
-- Query: [name]
-- Business Question: [what this answers]
-- SQL Techniques: [window functions, CTEs, etc.]
-- Redshift Notes: [anything that would differ from real Redshift]
-- Expected Output: [rows × cols shape]
-- Product Decision This Informs: [what a PM would do with it]
-- ============================================================
```

All queries must be **Redshift-compatible syntax where possible** and **actually run against the SQLite DB**. For functions that differ (e.g., `DATEDIFF` in Redshift vs SQLite's `julianday()`), include both versions commented out.

### `sql/01_engagement_metrics.sql`
- **Query 1:** DAU / WAU / MAU with 7-day rolling averages, stickiness ratio (DAU/MAU), day-over-day change using `LAG()`

### `sql/02_retention_cohorts.sql`
- **Query 2:** D1/D7/D14/D30 retention cohort matrix — users grouped by signup week, retention rate tracked across weeks. Self-join on user activity.

### `sql/03_content_performance.sql`
- **Query 3:** Engagement rate (likes+shares+comments / views) by language × content_category, with `PERCENT_RANK()` to rank categories within each language

### `sql/04_creator_analytics.sql`
- **Query 4:** Creator power-law analysis — cumulative share of engagement captured by top X% of creators using `NTILE(100)` and cumulative sums. Should show something like "top 1% of creators drive 40% of engagement."

### `sql/05_funnel_analysis.sql`
- **Query 5:** View → Like → Share → Follow funnel, overall and broken down by city tier. Drop-off rates at each stage using conditional aggregation.

### `sql/06_ab_test_analysis.sql`
- **Query 6:** A/B test readout computed entirely in SQL: per-group sample size, mean session duration, variance, standard error, z-statistic, approximate p-value. Include comment block explaining the statistical logic.

### `sql/07_monetization_analysis.sql`
- **Query 7:** Revenue per user, CTR, conversion rate — pivoted by city_tier × acquisition_channel using `SUM(CASE WHEN ...)`

### `sql/08_anomaly_detection.sql`
- **Query 8:** Anomaly detection on DAU — flag days where DAU deviates more than 2 standard deviations from the trailing 14-day mean, using window functions `AVG() OVER` and `STDDEV() OVER`. Cross-reference with festival flag to separate real anomalies from expected spikes.

### `sql/09_power_users.sql`
- **Query 9:** RFM-style user segmentation — Recency (days since last session), Frequency (sessions in last 30 days), Monetary (total watch time), each bucketed with `NTILE(5)`, combined into named segments (Champions / Loyal / At Risk / Hibernating / Lost)

### `sql/10_creator_retention.sql`
- **Query 10:** **Recursive CTE** identifying creators with consecutive-weekly-posting streaks. Show distribution of streak lengths.

### `sql/11_language_cross_analysis.sql`
- **Query 11:** Cross-language content consumption — do Hindi users watch Bhojpuri content? Build a language × language matrix of user-language vs. content-language consumption. Uses self-join and pivot.

### `sql/12_session_patterns.sql`
- **Query 12:** Session depth distribution — histogram buckets (0-1 min, 1-5 min, 5-15 min, 15-30 min, 30+ min), segmented by device_type. Uses `CASE WHEN` bucketing.

### `sql/13_festival_impact.sql`
- **Query 13:** Festival impact analysis — compare DAU, session duration, and engagement rate on festival days vs. non-festival days, with percentage lift and statistical significance check (basic t-test logic in SQL).

### `sql/14_device_segmentation.sql`
- **Query 14:** Device-tier behavior comparison — crash proxy (very short sessions <10 sec), app version distribution, engagement rate per device tier. Uses conditional aggregation and window functions.

### `sql/15_cohort_ltv.sql`
- **Query 15:** Cohort-based LTV proxy — for each signup-month cohort, compute cumulative watch time and ad revenue per user over 30/60/90 day windows. Chain of 3+ CTEs.
